# 사용자 위상 구분

> 이 문서에서 말하는 **위상(phase)** 은 순차적으로 올라가는 단계가 아니라,  
> **문제·환경·역할에 따라 전이되는 사고의 상태** 를 의미한다.  
> 동일한 사람도 상황에 따라 서로 다른 위상에 위치할 수 있다.

---

## 사용자 위상 (Usage Phase / Universal User)

### 할 수 있는 것의 본질

- 주어진 구조 안에서 **선택하고 최적화** 할 수 있다.

### 구조적으로 말하면

- 구조를 **사용** 할 수 있음
- 문제의 입력–출력 관계를 이해함
- 정해진 규칙·절차·프레임을 따라가며 조합함

### 대표 능력

- 설명된 규칙을 이해하고 적용
- 목표가 주어졌을 때 최선의 선택 탐색
- 비교·평가·효율 개선
- “이 중에 뭐가 더 나은가?”를 판단

### 이 위상의 핵심 문장

> 대부분의 사람은 이미 만들어진 틀 안에서 잘 선택하는 데 능하다.

### 특징

- 왜 이 구조가 생겼는지 몰라도 됨
- 다른 구조가 가능한지 몰라도 잘 작동함

---

## 경계 인식 위상 (Boundary Phase / Transitional User)

### 할 수 있는 것의 본질

- 구조를 **인식** 하고, 일부를 **수정** 할 수 있다.

### 구조적으로 말하면

- 구조를 **의식** 하기 시작함
- 규칙이 절대적인 것이 아님을 인지
- 구조의 **경계(boundary)** 를 감지함

### 대표 능력

- “이 규칙, 꼭 필요한가?”라는 질문
- 여러 프레임 비교
- 기존 방식이 안 맞는 상황 감지
- 국소적 구조 변경 (규칙 예외, 우회, 재조합)

### 특징적 상태

- 혼란과 통찰이 공존
- 설명 요구·질문 증가
- 기존 시스템에 어색함을 느낌

### 보편 사용자 관점에서 보면

> 어떤 사람들은 규칙을 쓰는 데서 그치지 않고, 규칙 자체를 의심하기 시작한다.

---

## 구조 설계 위상 (Structural Phase / Non-Universal User)

### 할 수 있는 것의 본질

- 구조를 **생성·재정의** 할 수 있다.

### 구조적으로 말하면

- 구조를 **대상** 으로 다룸
- 문제보다 문제의 **생성 조건** 을 봄
- 구조 간 변환과 통합이 가능

### 대표 능력

- 문제 자체를 다시 정의
- 프레임을 새로 만들거나 병합
- 보이지 않는 제약을 명시화
- 타인의 사고 구조를 읽고 대응

### 특징

- 설명보다 설계가 먼저 나옴
- 질문이 “무엇을 할까?”가 아니라  
  “왜 이게 문제로 보이지?”로 이동
- 타인에게는 추상적으로 보일 수 있음

### 보편 사용자 관점에서 보면

> 어떤 사람들은 게임을 잘하는 게 아니라, 게임의 규칙을 만든다.

---

## 요약

- **사용자 위상**: 주어진 틀 안에서 선택하고 최적화한다
- **경계 인식 위상**: 틀을 인식하고 일부를 조정한다
- **구조 설계 위상**: 틀 자체를 정의하고 생성한다

또는 일상적으로 표현하면:

> 사용자 → 조정자 → 설계자

---

## 중요한 구조적 주의점

- 위상은 **우열이나 서열이 아니다**
- 각 위상은 **필요한 환경과 책임 조건이 다르다**
- 모든 문제에 모든 위상이 동시에 필요한 것은 아니다
- **구조 설계 위상은 사용자·경계 인식 위상 없이 단독으로 유지되지 않는다**

이 문서는 능력 평가가 아니라,  
**역할과 사고 상태의 분화 구조** 를 설명한다.
