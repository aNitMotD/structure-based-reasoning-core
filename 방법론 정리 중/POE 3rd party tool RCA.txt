Impact Summary: Core Feature Broken & Persistence
Before diving into the technical analysis, I want to highlight the severity:
1. Core Feature Blocked: This issue completely renders the "Price Check" functionality unusable, which is the core value of this application.
2. Configuration Futile: I attempted to change the hotkey settings within the app to various other keys. The issue persists regardless of the key mapping. This confirms that Copilot's hook is aggressive and cannot be bypassed simply by changing user settings.
3. Confirmed on Stable Build: This critical defect exists in the Stable (Production) channel of Microsoft 365, meaning all standard users are potentially affected.
- Environment: I am using the Offline / Local Installation (Click-to-Run) version. While I encountered this on the local install, the behavior suggests an OS-level background hook that could potentially affect any version employing a similar resident process.
- Example Addition: The issue strictly occurs when the game client is running and has the window focus, triggering the third-party application's global hotkey registration.

I identified this behavior in late October and filed a formal report via Microsoft Feedback Hub. After performing a deep-dive RCA locally, I can confirm this is caused by Microsoft 365 Copilot aggressively hooking and mutating OS input events.

1. The Mechanism (Broad Input Mutation) I analyzed the debug logs using uIOhook and found a critical "Phantom Injection" behavior. This is not limited to a single key: * Wildcard Interception: Copilot intercepts not just Ctrl + D, but a wide range of combinations including Ctrl + Alt + D and Ctrl + Shift + D. None of these are official Copilot hotkeys, yet they trigger the Copilot Message Window. * Persistence: Even when I remapped the app's hotkey to something completely different (e.g., Shift + Alt + G), Copilot STILL intercepted the input and forced its window open.

Mutation: In all cases, Copilot CONSUMES the user input (making it disappear for the app) and INJECTS its own invocation key (Ctrl + Alt + C) into the OS.

Note on Testing Methodology

Although the primary use case for this tool is hovering over items for price checking, the screenshots above were taken in the character selection screen to provide a cleaner background for the log overlay visibility.
I have already confirmed that the exact same input mutation occurs when hovering over items in-game. The context (in-game vs. menu) does not affect the MS365 Copilot hook's aggression.

 
2. Code Verification I reviewed Shortcuts.ts in this repo. The logic for registering/unregistering shortcuts based on window focus is implemented correctly ("Good Citizen" pattern). The issue is NOT within Awakened PoE Trade's codebase. It is an architectural defect in the Windows Copilot process.

3. Why PowerToys / Remapping Fails A common suggestion is to use PowerToys to remap or disable the Copilot shortcut. However, this DOES NOT fix the issue.

Since Copilot consumes the original Ctrl + D event before injecting its own key, the original input is already lost by the time PowerToys sees the injected key.
Remapping only stops the Copilot window from popping up (visual fix), but the application still fails to receive the hotkey command (functional failure).
4. The Solutions (Workarounds) Since this is an OS-level conflict caused by MS, the reliable fix is to prevent the Copilot process from launching and hooking keys:

Method A (Recommended): Disable Copilot in Windows Settings > Apps > Startup (or Task Manager > Startup apps). This ensures the process doesn't load its hook during system boot.
Method B (On-Demand Use): If you specifically need Copilot features, launch the app manually only when needed, and terminate (quit) the process immediately after use. Do not leave it running in the background while playing.
5. Note on Broader Ecosystem Impact
Potential Impact on Other Electron Apps It is highly likely that this issue is not isolated to Awakened PoE Trade. Since this conflict stems from how MS365 Copilot aggressively interacts with the OS input stack versus how the standard Electron globalShortcut API registers hooks, any other Electron-based application running in the background that relies on similar global hotkeys (especially those involving Ctrl modifiers) is likely to experience the same "Silent Failure" or input mutation.

This appears to be a systemic conflict between the MS365 Copilot service and the Electron framework itself on Windows.

Systemic Conflict with Other Hotkey Registrants (Unverified Question): The core conflict arises at the OS hook layer upon hotkey registration. While direct evidence is confirmed for Electron-based applications, a critical question remains: Does this aggressive hooking mechanism affect applications built using other frameworks (e.g., C++, Python, .NET) that register global hotkeys using the OS-level API? Given the root cause is the high priority of the Copilot hook, it is highly probable that all standard hotkey registrations are vulnerable, regardless of the application's underlying technology.
